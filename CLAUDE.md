## Architecture

### Current state: UI prototype, no backend

This codebase was generated by v0.app (`generator: 'v0.app'` in `app/layout.tsx`) and is a **visual prototype only**. There is no file upload handling, no API routes, no database, and no real analysis pipeline. All data in every component is hardcoded (`mockCases` in `PhaseIntake.tsx`, fixed EXIF/geo values in `PhaseAnalysis.tsx`, a fixed 94/100 score and findings text in `PhaseReport.tsx`, canned log lines in `KernelTerminal.tsx`). `AuthScreen` accepts any input and logs in unconditionally. `PhaseAnalysis` auto-advances to the next phase via a hardcoded `setTimeout(..., 3000)` rather than reacting to real processing completion.

When implementing real features, treat every visible number/string in these components as a placeholder to be replaced by real state, not as a spec to preserve literally.

### Single-page, state-driven workspace (no routing)

The entire app is one route (`app/page.tsx`). There is no App Router page-per-view; instead a `CurrentView` union type (`'auth' | 'intake' | 'idle' | 'analysis' | 'explorer' | 'report' | 'archive' | 'status' | 'settings'`) held in `useState` at the top of `page.tsx` decides which component renders in the main pane. `Sidebar` and `TopHeader` both take `currentView`/`onViewChange` and drive navigation by calling back up to this same state — there is no URL sync, so browser back/forward and deep-linking to a specific view do not work.

The intended workflow order is: `auth → intake → analysis → explorer → report → archive`, with `idle`, `status`, and `settings` as side views reachable from the sidebar at any time.

Layout shell is fixed across all views: `Sidebar` (left nav) + `TopHeader` (breadcrumb/title bar) + main scrollable content + `KernelTerminal` (bottom log panel), assembled once in `page.tsx` and not per-view.

`components/TopBar.tsx` is an orphaned component — not imported anywhere. It appears to be a leftover from an earlier layout iteration; don't assume it's part of the current UI.

### Data flow between phases does not exist yet

No investigation/case object is threaded through the phase components — each one independently owns and renders its own mock data. Anyone wiring up real functionality needs to introduce a shared state layer (e.g. lifted state or a store) for the in-progress investigation (file, hash, extraction results, findings, score) that flows from `PhaseIntake` through to `PhaseReport`, since none of that plumbing currently exists.

### Styling system

- Tailwind v4, CSS-first config — there is no `tailwind.config.js`; the theme (colors, radii, sidebar/chart tokens) is defined via `@theme inline` and CSS custom properties directly in `app/globals.css`.
- shadcn/ui is configured via `components.json` (style: `base-nova`, base color `neutral`, RSC + TSX on), but only one primitive has actually been generated so far: `components/ui/button.tsx`. Use the shadcn CLI to add more primitives rather than hand-rolling them, to stay consistent with the configured style.
- Visual identity is a dark "forensic terminal" theme: near-black background (`#0B0C0E`), zinc/amber palette, monospace type throughout (`font-mono` applied broadly), sharp corners (`rounded-none` used deliberately instead of the shadcn default radius) — preserve this aesthetic (no rounded corners, no light theme) when adding new UI.
